Validation UX cross-field

Quando un commit individuale viola una regola che richiede altri campi (es. "evento referenziato precede vita.nascita del personaggio"), come reagisce l'UI? Hard reject, auto-prompt, o soft validation?

Modello di validation

Grammatica
Tap fuori → il dominio prova a committare → cross-validation fallisce.
Field non lascia l'edit state. Border diventa rosso. Toast in basso al detail pane spiega l'errore.
L'utente DEVE correggere il valore o annullare con ESC. Tap fuori non funziona finché lo stato non è valido.
Tutti gli altri field rimangono disabilitati per safety finché questo non è risolto.
Garanzia totale: l'aggregate non è mai in stato invalido
Errore visibile e immediato
Anti-flow: l'utente deve risolvere il vincolo PRIMA di poter fare altro. Su iPad, dove il pensiero è non-lineare, è frustrante
"Tutti gli altri field disabilitati" rompe il paradigma inline per-campo
Cross-aggregate hard reject = serve sempre la lock dell'altro elemento — concorrenza problematica con CRDT
Anti-Notion / anti-Apple Notes — nessuna app inline per-campo lavora così
Su sync offline-first (Jazz CRDT), un peer remoto può creare uno stato "valido localmente, invalido after merge" — l'hard reject non sa cosa fare

Validation gira post-commit → se trova un'incoerenza, popover HeroUI compare ancorato al field problematico.
Due azioni: "Correggi data" (re-apre il field in edit) o "Lascia stare" (dismiss popover, warning resta come icona inline).
Il popover è dismissibile da tap fuori.
Commit non è bloccante — flow inline preservato
Suggerisce attivamente la correzione → educa l'utente sul vincolo
Dismissible → utente può ignorare se sa cosa sta facendo
Modal-ish chrome: ogni commit con warning fa apparire un popover. Su edit non-lineari (15 modifiche di fila) diventa rumoroso
Z-index war con altri popover (collegamento picker, ecc.)
"Lascia stare" fa restare il warning ma l'utente l'ha visto — può sentirsi un nag
Su Jazz sync, un peer remoto può scatenare popover imprevisti quando l'altro elemento cambia

Grammatica
Tap fuori → commit sempre ok. useElemento(id) ritorna sempre { elemento, warnings }.
4 livelli di propagation della validation:
  1. Inline icon ⚠ accanto al field problematico (24×24, hover/tap → tooltip)
  2. Header badge "2 da rivedere" pill warning sul detail pane
  3. Sidebar marker ● (8px warning dot) accanto al nome elemento nella lista
  4. Validation panel espandibile in fondo al detail pane con la lista warnings cliccabili (jump al field)
Nessuno dei 4 livelli blocca l'utente — sono passive cues.
Zero friction sul flow — commit sempre accettato, edit non-lineare libero
Discoverable a 4 livelli — l'utente vede il problema da qualunque pane
Pattern DDD canonico: invariants check at read, not at write (Eric Evans)
Compatibile con Jazz CRDT sync — peer remoto può cambiare un campo, la validation gira automaticamente sul re-render
Pattern presente in Notion ("missing required property"), Linear ("issue blocked"), Things 3 ("waiting on")
Validation panel = jump-to-fix list — UX da debugger
L'utente può ignorare i warning indefinitamente → serve un report "elementi incompleti" a workspace level (futuro plan)
Re-validation a ogni read può essere costoso su workspace grandi — memoizzare con React Compiler / Legend State
Note iPad-specific
  • Strategia ibrida con sketch 5: i composite component (Vita, Regno, Periodo, Intervallo<T>) eliminano cross-validation intra-aggregate by construction. La soft validation D si occupa solo di cross-aggregate (es. evento referenziato precede vita.nascita) — il sottoinsieme di errori si riduce drasticamente.
  • HeroUI mapping: warning icon è <TriangleAlert> da lucide, badge header è <Chip color="warning" variant="flat" size="sm">, validation panel è <Card variant="bordered"> con classe warning.
  • Sidebar marker: 8×8px dot warning. Discreto in idle, immediato da scansionare visivamente quando l'utente cerca "cosa devo sistemare".
  • Tap su warning icon: mostra tooltip su hover (desktop review) o tap (iPad). HeroUI <Tooltip> primitiva, dark theme, anchor al field-value.
  • Workspace report (futuro): dashboard "elementi da rivedere" filtrabile per workspace — fuori scope di S02/R005, candidato per S04+.
  • Performance: validation gira in useElemento hook, memoizzato. React Compiler abilitato (Constitution IX).
