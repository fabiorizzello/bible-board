Markdown descrizione — Milkdown

Abraamo ha una descrizione markdown con grassetto, link a Isacco, e una lista. Quale chrome (UI) montiamo sopra Milkdown per editarla senza rompere l'inline per-campo?

Nascita 2000 a.E.V.
Grammatica
Tap nel campo descrizione → border teal + focus ring (HeroUI Input lg).
Idle (no selezione): zero chrome, "sembra solo testo formattato".
Su selezione testo (tap-hold + drag handle iPad nativo): bubble menu floating con bottoni 44×44 — Bold / Italic / Strike / Heading / Link / Code.
Inline shortcuts markdown sempre attivi (**bold**, *italic*, [link](url)) auto-convertiti da Milkdown.
Tap fuori → commit (sketch 1A).
Chrome zero in idle — perfettamente coerente con inline per-campo
Touch target bubble 44×44 — usabile anche senza tastiera
iPad supporta nativamente la selezione testo (tap-hold + handles + magnifier loupe)
Milkdown ha plugin @milkdown/plugin-tooltip per il bubble — implementazione diretta
Mental model familiare (Notion, Linear docs, Apple Notes recente)
Block-level (heading, lista, quote) richiede o markdown shortcut o tap su Heading nel bubble (meno discoverable di un menu dedicato)
Selezione touch è meno precisa del cursor: serve attenzione su drag handles iPad

Grammatica

Tutti i tool 44×44 sempre visibili: Bold / Italic / Strike / H1 / H2 / List / Numbered / Link / Quote.
Active state riflette il contesto del cursore.

Discoverability massima — tutti i tool visibili da subito
Heading e lista con un tap — zero shortcut markdown da imparare

Toolbar 60px occupa verticale prezioso sul detail pane (iPad portrait soprattutto)
Incoerente con gli altri field inline (nome, data, tipo) che non hanno chrome — l'utente vede "due paradigmi"
Sembra "una webapp wordpress", non un'app iPad-native

Grammatica
Tap nel campo → border teal, ma il campo sembra identico al view.
Inline: shortcut markdown auto-convertiti (**bold**, *italic*, [link](url)).
Block-level: digita / → popover (HeroUI Listbox) con lista, heading, quote, code.
Row 48px nel menu, target iPad-friendly.
Tap fuori → commit.
Chrome assolutamente zero — densità massima, coerenza totale con altri field inline
Slash menu è popover (stesso primitivo del collegamento picker — coerenza interna)
Power user con Smart Keyboard godono — tutto da tastiera
Anti-iPad-touch-only: senza tastiera, l'utente deve aprire la softkeyboard solo per digitare / — friction enorme
Discoverability bassa: senza onboarding nessuno sa che / esiste
Bold / Italic richiedono shortcut markdown: nessun bottone visibile per non-power-user
Apple Pencil + dita come input primario → questo design li penalizza
Note iPad-specific
  • Editor locked: Milkdown (MIT, headless, ProseMirror). Lo sketch confronta solo la UI chrome sopra Milkdown — l'engine è già deciso.
  • Data model invariato: Elemento.descrizione: string resta markdown plain text. Milkdown serializza/deserializza al boundary adapter; il dominio non vede MDAST.
  • Plugin Milkdown necessari: @milkdown/plugin-tooltip (bubble), @milkdown/plugin-slash (slash menu), @milkdown/preset-commonmark, @milkdown/plugin-listener (commit on blur).
  • Touch target: bubble button 44×44, sticky toolbar button 44×44, slash menu row 48×280px. Apple HIG + WCAG 2.5.5.
  • Variante ibrida (futura): A+C — bubble menu su selezione (inline) + slash menu su / (block-level) — coprirebbe entrambi i casi mantenendo chrome zero. Da valutare in plan 02-03 se R005 ci arriva.
  • Coerenza: commit blur-to-save (sketch 1A), border teal e focus ring identici al pattern degli altri field, no mode swap.
