Composite Vita component

Coppia naturale nascita + morte (opt) come componente unico. Risolve la cross-validation per costruzione: morte senza nascita = stato impossibile by design (Parse, Don't Validate).

Modello di dominio
// shared/value-objects.ts type Vita = { nascita: DataStorica; morte?: DataStorica; // opzionale by construction }; // constructor enforces invariant: morte (se presente) > nascita const creaVita = ( nascita: DataStorica, morte?: DataStorica ): Result<Vita, VitaError> => morte && morte.compareTo(nascita) <= 0 ? err({ tipo: 'morte_prima_di_nascita' }) : ok({ nascita, morte }); // Personaggio aggregate type Personaggio = { ... vita?: Vita; // o assente, o valida — terzo stato impossibile };

Densità massima — una sola row anche per due valori
Inline per-campo preservato — half editabili indipendentemente, no mode swap
Cross-validation by construction: morte senza nascita o morte ≤ nascita = stato impossibile, no warning UI da gestire
Calcolo derivato visibile (175 anni) senza chrome — chip muted a destra
Generalizzabile a Intervallo<DataStorica> per Regno, Periodo, Profezia, ecc.
Su date molto lunghe (es. "fine III secolo a.E.V.") la row può wrappare — gestire wrap o truncate
Tap precision: i due half sono stretti (~140px), serve attenzione hit-area

Note iPad-specific
  • Generalizzazione: il pattern composite serve anche per Regno (re), Periodo storico, Profezia (enunciazione → adempimento), Evento esteso. Tutti istanze di un VO Intervallo<DataStorica> riusabile. Da decidere in plan 02-03.
  • Parse, Don't Validate: il costruttore creaVita() rifiuta morte ≤ nascita. Lo stato invalido non esiste mai a livello dominio. Coerente con CLAUDE.md V-bis.
  • Soft validation D residua: i composite eliminano cross-validation intra-aggregato. La strategia D (sketch 6) resta utile per cross-aggregato (es. "evento referenziato precede vita.nascita del personaggio").
  • HeroUI mapping: ogni half è <Input variant="bordered" size="lg"> in stato edit; in idle è solo testo (zero chrome).
  • Coerenza: commit blur-to-save (sketch 1A), nessun bottone esplicito, focus ring HeroUI standard.
