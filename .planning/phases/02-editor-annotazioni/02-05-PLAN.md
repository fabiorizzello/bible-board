<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User -> rewritten `ElementoEditor` | The user can input arbitrary text into title, description, tags, year fields, origin, tribe, and link-add flows. |
| Rewritten shell -> `workspace-ui-store` session overlay | The UI writes only to in-memory session overrides and field-surface state; no persistence or server boundary is added by this rewrite. |
| `ElementoEditor` commit helpers -> `normalizeElementoInput` | Domain normalization remains the single validation boundary for normalized fields before they enter the session overlay. |
| Local link/add flows -> session overlay patch | Link add/remove flows still require special patch writes; they must not bypass the existing overlay model. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05-01 | Tampering | rewritten field commit flows | mitigate | Keep `normalizeElementoInput` as the single normalized write gate. Every scalar/type-specific commit builds a next `ElementoInput`, normalizes it, and only then calls `commitNormalizedElement`. |
| T-02-05-02 | Tampering | link add/remove drawers | mitigate | Keep `commitElementPatch(element.id, { link: ... })` constrained to explicit add/remove flows. Do not introduce generic object spread writes from uncontrolled drawer state. |
| T-02-05-03 | Information Disclosure | warning/review drawer | accept | Warnings remain UX-only and describe missing/soft data issues. They must not reveal hidden ids, internal errors, or stack traces. |
| T-02-05-04 | Denial of Service | lazy Milkdown editor mount | mitigate | Keep Milkdown lazy-mounted only when editing `descrizione`, and close back to prose preview on commit/close so the heavy editor is not always mounted. |
| T-02-05-05 | Spoofing | deferred S03 affordances | mitigate | Any S03 source-editing affordance shown in the rewrite must be visibly marked deferred and must not open a working editor path that suggests functionality not actually delivered. |
| T-02-05-06 | Elevation of Privilege | duplicate pane/fullscreen editor logic | mitigate | Keep one shared editor shell component so fullscreen cannot drift into a second, less constrained editing path. |
| T-02-05-07 | Repudiation | toast-only feedback | accept | The prototype remains client-side and session-only. Toast feedback is sufficient for S02; durable audit trails belong to later persistence phases. |

The rewrite changes UI structure, not trust boundaries. Security posture remains bounded by the existing client-side prototype model, and the main requirement is not to create a second unchecked write path while replacing the editor shell.
</threat_model>
